El escenario

💻

Llevas 2 horas concentrado en tu feature. Todo avanza. Estás en flujo.

☕

Archivos modificados, lógica a medio escribir, varios cambios locales sin commit.

🚨

Tech Lead — Teams

"Hay un bug en producción. Es urgente. ¿Puedes verlo ya?"

😰

Tu feature no está terminado. Tienes cambios sin commit. ¿Qué haces?

El flujo más común

terminal 
# Estás en feature/login con cambios sin commit...
git stash                  # guardar cambios temporalmente
git checkout main
git pull origin main
git checkout -b hotfix/bug
# ... arreglas el bug ...
git push origin hotfix/bug
git checkout feature/login
git stash pop              ⚠ posible conflicto
🤷

¿Cómo lo solucionarías hoy?

La respuesta más común: git stash

¿Qué hace?

Guarda tus cambios sin commit en una pila temporal y te deja la rama "limpia" para poder cambiar.

¿Cuáles son sus limitaciones?

→

Cada cambio se guarda con un índice: stash@{0}, stash@{1}... difícil hacerle seguimiento

→

Si la rama avanzó, stash pop puede generar conflictos

→

Aún así tuviste que cambiar de rama — el cambio de contexto ya ocurrió

El costo real

La interrupción va a ocurrir de todos modos.
Lo que worktree evita es sumar fricción técnica:

→

Guardar y restaurar cambios manualmente

→

Reconstruir el estado del directorio al volver

→

Resolver conflictos que no tenían por qué existir

¿Y si no tuvieras que
cambiar de rama?

¿Y si pudieras abrir otra carpeta,
con otra rama, sin tocar nada de lo que estás haciendo?

Concepto

Git Worktree

Otro directorio de trabajo. Otro HEAD.
El mismo repositorio.

📁 Cada uno con su propio HEAD
🔗 Comparten objetos y refs
⚡ No requiere descarga
🧩 No reemplaza stash, lo complementa

Modelo mental

📁
mi-proyecto/
main
📁
mi-proyecto-feature/
feature/login
📁
mi-proyecto-hotfix/
hotfix/bug-123
.git

Un solo repositorio

Como escritorios virtuales del sistema operativo, pero para git.

Un poco de historia

2015
Git 2.5 — Se introduce git worktree con add, prune y list.

Contribución principal: Nguyễn Thái Ngọc Duy. Se marca como experimental.

2017+
Evolución continua — Se agregan lock, move, remove y repair en versiones posteriores.
2026
11 años disponible. Funcional y maduro, aún con advertencia oficial sobre submódulos.

Worktree vs Clone vs Stash

Stash Clone Worktree
Propósito Pausar cambios temporalmente Repositorio independiente Contextos activos en paralelo
Cambio de contexto Sí — sales de tu rama No (otro directorio) No — cada carpeta es independiente
Tu trabajo en progreso Se pausa (guardado en pila) Intacto Intacto — sigue activo en su carpeta
Descarga objetos No Normalmente sí No
Directorio adicional No Sí (independiente) Sí (vinculado)
Conflicto al restaurar Posible No aplica No aplica
Configuración y remotes Compartidos Independientes Compartidos

Stash = pausar y reanudar en el mismo lugar. Worktree = dos lugares activos al mismo tiempo, sin interrumpir ninguno.

Internals

Antes de ir al "cómo"...

Entendamos qué pasa por debajo.
No es magia — es una estructura de archivos.

Anatomía de .git/

objects/ Blobs, trees, commits — toda la historia
refs/ Punteros a commits (ramas, tags)
HEAD ¿En qué rama estás ahora?
index El staging area (lo que va en el próximo commit)
worktrees/ ← Aquí ocurre la lógica de worktree

¿Qué pasa cuando haces worktree add?

1
En el repo principal:

Git crea .git/worktrees/<nombre>/ con su propio HEAD e index

2
En la carpeta nueva:

Recibe un archivo .git (no carpeta) que apunta de vuelta al principal

3
Lo compartido:

Objetos, remotes, la mayoría de refs y la configuración del repositorio siguen siendo compartidos

Reglas y restricciones

🚫

Ramas únicas (por defecto)

Git evita que una misma rama esté activa en dos worktrees. Protege contra modificaciones concurrentes del mismo HEAD.

🏠

Main worktree protegido

git worktree remove solo elimina linked worktrees. El main worktree no se puede eliminar con ese comando.

🔗

Compartido

Objetos, la mayoría de refs (ramas, tags), remotes, configuración del repositorio.

🔒

Independiente por worktree

Working directory, HEAD, index (staging area) y el estado de operaciones como merge o rebase.

Comandos

Todo lo que necesitas

git worktree add ../ruta rama

Crear un nuevo worktree

git worktree add ../ruta -b nueva base

Crear worktree con rama nueva basada en otra

git worktree list

Ver todos los worktrees activos

git worktree remove ../ruta

Eliminar un worktree (la rama sigue existiendo)

git worktree prune

Limpiar referencias a worktrees que ya no existen en disco

Casos de uso reales

🔥
Hotfix urgente

Arreglar un bug sin perder el progreso de tu feature

👀
Code review

Revisar un PR sin salir de tu rama actual

🧪
Tests largos

Correr tests en una rama mientras desarrollas en otra

🔀
Comparar ramas

Dos carpetas lado a lado para ver diferencias de comportamiento

🤖
Agente IA en paralelo

Delegar una tarea a un agente en su propio worktree mientras tú sigues trabajando

¿Cómo paso mi feature a main?

El merge/PR no cambia. Los worktrees son carpetas — el merge es de ramas.

Opción A: Merge desde el worktree de main

cd ../mi-proyecto-main && git merge feature/login

Opción B: Push + PR (lo más común)

git push origin feature/login → abrir PR en GitHub

Punto clave:

Las ramas existen en el repositorio. Los worktrees son solo ventanas a esas ramas. No hay que "pasar" nada de un worktree a otro.

¿Cuándo no compensa?

⚠️

Si la interrupción es breve (2 min) y un stash simple resulta suficiente

⚠️

Si dos ramas compiten por los mismos recursos locales: puertos, bases de datos, colas, Docker

⚠️

Si el proyecto requiere setup costoso por carpeta (mvn install, npm install, builds pesados)

⚠️

Si se necesita aislamiento fuerte de configuración o credenciales — ahí un clone o contenedor es mejor

¿Qué se comparte? La historia de Git (.git/objects, commits, refs).
¿Qué NO se comparte? Archivos de trabajo, dependencias, builds, procesos.
